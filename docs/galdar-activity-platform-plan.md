# 갈다 활동 기록 플랫폼 기획안

> 갈다가 운영하는 모든 사업의 "기록·축적·발행"을 담당하는 멀티 프로젝트 웹앱.
> 첫 적용 프로젝트는 **2026 국립과천과학관 \<과학과애\>**.
> 기존 생태도감 앱(eco-explorer)은 그대로 운영하고, 코드를 복제한 새 리포지토리에서 개발한다.

---

## 1. 컨셉: 사업별 앱이 아니라, 프로젝트를 담는 플랫폼

지금까지는 사업(우도 생태탐사 등)마다 앱을 새로 만들거나 고쳐 썼다. 앞으로는 **"갈다 활동 기록 앱" 하나에 클라이언트/사업 단위로 프로젝트를 개설**해서 쓴다.

```
갈다 활동 기록 플랫폼
├── 프로젝트: 과학과애 2026 (국립과천과학관)   ← 첫 적용
├── 프로젝트: ○○ 생태탐사 2026                ← 향후
├── 프로젝트: △△기업 과학문화 프로그램        ← 향후
└── ...
```

프로젝트마다 독립적으로 갖는 것:

- **공개 사이트**: `/p/[프로젝트-slug]/...` — 프로젝트 이름·로고·색상, 프로그램 안내, 갤러리
- **프로그램/회차 구조**와 참가자 기록
- **스탭 멤버십**: 어떤 스탭이 이 프로젝트에서 기록/발행 권한을 갖는지
- **발행 채널 설정**: 인스타 계정, 블로그, 기본 해시태그, 채널별 글 톤(클라이언트 채널 / 갈다 채널)
- **모듈 ON/OFF**: 아래 모듈 중 이 프로젝트에 필요한 것만 켠다

### 기능 모듈 (프로젝트별 선택)

| 모듈 | 내용 | 출처 |
|---|---|---|
| **안내 홈** | 프로그램 캘린더·상세 페이지·신청 링크(LMS 등 외부) | 신규 |
| **도감** | 사진 → AI 종/장소 인식 → 도감 뷰(모아/선택/지도) | eco-explorer 이식 |
| **탐방 기록** | 사전 자료·사전 질문 → 현장 기록 → 탐방별 갤러리 | eco-explorer 이벤트 모델 확장 |
| **결과물 갤러리** | 참가자 작품/결과물 업로드, 좋아요, 랭킹 | eco-explorer 이식 |
| **기록·발행** | 스탭 사진+메모 → AI 포스팅 초안 → 검토 → 발행 | 신규 (핵심) |

같은 기능을 다른 사업에서도 재사용할 수 있도록, 모듈은 프로젝트 설정만 바꾸면 동작하게 만든다 (코드 분기 금지, 설정 데이터로 제어).

---

## 2. 기존 생태도감 앱과의 관계

- **기존 앱(galdar-eco-explorer)은 손대지 않는다.** 우도 데이터·URL·운영 모두 현행 유지.
- 리포지토리를 **복제(fork)해서 새 리포지토리/새 배포**로 시작한다. 검증된 코드(업로드, AI 분석, 갤러리, 좋아요, 지도)를 재사용하면서 멀티 프로젝트 구조로 리팩터링.
- 기존 데이터 이관은 하지 않는다. 필요해지면 우도 사업을 프로젝트 하나로 이관하는 것은 나중에 별도 작업.

## 3. 참가자용 (프로젝트 공개 사이트)

로그인 없음(현행 editToken 방식 유지), 회차별 QR로 진입하는 동선이 기본 원칙.

### 공통 (모든 프로젝트)

- 프로젝트 홈: 월별 일정 캘린더, 분야 필터, 프로그램 상세(가이드 페이지), 신청 링크
- 회차 QR → 해당 회차 페이지(기록 업로드 / 자료 / 설문 링크)
- 기록 갤러리: 분야별/회차별 모아보기, 좋아요, 내 기록 수정·삭제

### 과학과애 프로젝트에서의 모듈 구성

| 분야 | 사용 모듈 | 비고 |
|---|---|---|
| 시민과학 | 도감 | "과천 지역 생태도감". 매미 지도·새 지도·외래생물 지도 등 세부 프로젝트를 phase로 운영. 위치 카테고리를 과천 환경(하천/공원/산림/주택가)으로 조정 |
| 현장과학 | 탐방 기록 | 탐방 1회 = 1단위. 사전 질문 폼(사이언스 프리퀄 소재) → 현장 사진+메모(통신 불량 지역 대비 업로드 재시도 큐) → 탐방 갤러리를 과학과애 라운지 콘텐츠로 재사용 |
| 실험·예술과학 | 결과물 갤러리 | 바이오아트·드로잉 등 결과물 업로드. 연속 강좌(과학관 드로잉)는 회차 누적으로 "작은 성장" 가시화 |
| 전체 | 안내 홈 + 기록·발행 | 접수/결제/수료증은 과학관 LMS 담당이므로 앱 범위 제외 |

## 4. 스탭용

### 4.1 핵심: 기록 → 발행 파이프라인

스탭 입력은 **사진 여러 장 + 짧은 메모 + 초상권 동의 확인 체크** 뿐.

```
[기록]  프로젝트 → 회차 선택 → 사진 업로드 + 메모(하이라이트, 참여자 반응, 강사 멘트)
        → 초상권 동의 체크 (미체크 시 발행 단계 진입 불가)
[생성]  AI(Claude API)가 회차 메타데이터(프로그램명·강사·일시, 시트에서 동기화된 정보) + 메모 + 사진으로
        ① 블로그 후기 초안 (제목 + 본문 + 사진 배치 위치, 800~1200자)
        ② 인스타 캡션 (후킹 문장 + 본문 + 프로젝트 설정의 해시태그)
        를 채널별 톤(클라이언트 채널/갈다 채널)으로 생성
[검토]  스탭이 수정 → 승인 (AI 생성물은 반드시 사람 검토 후 발행)
[발행]  1단계: 원클릭 복사 + 리사이즈 사진 묶음 다운로드 (반자동, 즉시 가능)
        2단계: 인스타 자동 발행 (Meta Graph API, 비즈니스 계정 + FB 페이지 연동 필요)
        3단계: 네이버 블로그 — 공식 글쓰기 API가 사실상 중단 상태라 반자동이 기본
```

발행 상태(초안/검토중/발행완료)와 URL을 기록해 **월별 의무 건수(과학과애: 블로그 2건+, 영상 1건) 달성 현황**을 프로젝트 대시보드에서 추적.

### 4.2 프로그램 관리 — 구글 시트가 운영 원장

프로그램 정리는 이미 구글 시트로 하고 있으므로, 앱에서 이중 입력하지 않는다.

- **1단계: CSV 가져오기** — 시트를 CSV로 내보내 admin에서 업로드하면 프로그램/회차가 생성·갱신됨 (프로그램명+월 기준 upsert, 또는 시트에 ID 컬럼 추가 권장)
- **2단계: 시트 동기화** — Google Sheets API(서비스 계정에 시트 공유)로 "동기화" 버튼 한 번에 반영
- 시트 컬럼 ↔ 앱 필드 매핑은 프로젝트 설정에 저장 (프로젝트마다 시트 양식이 달라도 대응)

#### 과학과애 시트 실제 구조와 매핑 (2026.6 확인)

**탭 1 — 프로그램 목록** (월별 그룹, 총 83회차 / 실참여자 845명 집계):

| 시트 컬럼 | 앱 필드 | 비고 |
|---|---|---|
| 분류 (시민과학/실험과학/예술과학/현장과학/소모임/특별) | `Program.category` | 정규 4분야 외에 **소모임·특별** 카테고리 존재 → enum이 아닌 자유 카테고리로 설계 |
| 프로그램명 | `Program.title` | 머리의 ★(기획안 작성/확인 완료)·☆(기획안 페이지 작업 완료)는 제목에서 분리해 `Program.planStatus`로 |
| 진행일자 / 시간 | `Session.dateTime` | "7/16, 7/23, 7/30, …"처럼 한 행에 여러 날짜 = 다회차 → 행 하나를 여러 Session으로 전개. "8.1(토) 혹은 8.2(일)" 같은 미확정 표기는 가져오기 시 경고 처리 |
| 장소 (교육관/전시관/패밀리/로비/천체투영관 등) | `Session.place` | |
| 대상 (가족동반/교사 소모임/꽃중년 등) | `Program.audience` | |
| 참가비 (무료/1만원/2만원/3만원) | `Program.fee` | |
| 진행자 | `Program.instructor` | 복수 진행자(쉼표 구분) 지원 |
| 연락처 / 이메일 | 가져오지 않음 | 개인정보 — 앱 DB에 저장하지 않는 것을 원칙으로 |
| 회차 / 정원 / 실참여자 | `Program.totalRounds` / `Session.capacity` / `Session.attendance` | 실참여자는 운영 후 기입 → 역방향(앱→시트) 동기화 후보 |
| 비고 | `Program.note` | |

**탭 2 — 주간 캘린더** (요일×시간대 그리드, "프로그램명/진행자/장소"): 데이터 원본이 아니라 탭 1의 파생 뷰 → 가져오기 대상에서 제외. 오히려 앱의 안내 홈 캘린더가 이 수작업 탭을 대체하는 것이 목표.

### 4.3 운영 보조

- **스탭 계정**: 이메일 매직링크 수준의 가벼운 인증. 롤: recorder(기록) / publisher(발행 승인) / admin(프로젝트 설정). 프로젝트별 멤버십으로 권한 분리
- **초상권 동의 관리**: 회차별 전자 동의 폼(QR) 또는 서면 수합 체크. 프로젝트 종료 후 파기 절차 지원
- **통계 대시보드**: 회차별 기록 수·좋아요·도감 종 수·포스팅 현황 → 과학과애의 월 2회 과학관 보고에 바로 활용
- **모더레이션**: 제출물 숨김/일괄 태깅 등 기존 admin 기능 유지

## 5. 데이터 모델 (Prisma, 멀티 프로젝트)

```
Project        id, slug, name, clientName, branding(JSON), modules(JSON), channelConfig(JSON),
               sheetMapping(JSON), isActive
StaffUser      id, email, name
Membership     staffUserId, projectId, role(recorder|publisher|admin)

Program        id, projectId, category, title, description, instructor, externalId(시트 ID), applyUrl
Session        id, programId, round, dateTime, place, capacity, qrSlug, surveyUrl

Submission     (기존 유지) + projectId, sessionId    // 참가자 기록·도감·결과물
Like           (기존 유지)
Event/Phase    (기존 유지, projectId 추가)            // 도감 프로젝트·탐방 단위로 계속 사용

StaffLog       id, sessionId, authorId, memo, mediaUrls(JSON), consentConfirmed, createdAt
GeneratedPost  id, staffLogId, channel(blog|instagram), tone, title, body, hashtags,
               status(draft|approved|published), publishedUrl, publishedAt
```

모든 공개 쿼리는 `projectId` 스코프 필수 — 프로젝트 간 데이터가 섞이지 않게 하는 것이 멀티 프로젝트의 제1원칙.

## 6. 로드맵 (과학과애 일정 기준: 공모 6월, 정규 6~11월)

| 단계 | 시점 | 내용 |
|---|---|---|
| **Phase 0** | 즉시 | eco-explorer 복제 → 새 리포지토리/배포 생성, Project 스코프 도입 리팩터링 |
| **Phase 1** | 6월 시민과학 특강 전 | 과학과애 프로젝트 개설: 안내 홈 + 과천 도감 + 스탭 기록 화면 + AI 초안 생성(반자동 발행) + CSV 가져오기 |
| **Phase 2** | 7~8월 | 탐방 기록 모듈(사전 질문·오프라인 재시도 큐), 인스타 자동 발행, 스탭 계정 인증, 시트 자동 동기화 |
| **Phase 3** | 9월~ | 통계 대시보드, 결과물 갤러리 고도화, 초상권 전자 동의 폼, 두 번째 프로젝트 온보딩으로 멀티 프로젝트 검증 |

기술 스택 유지: Next.js + Prisma/Postgres + Vercel Blob + Gemini(이미지 분석) + Claude API(포스팅 초안) + Google Sheets API(원장 동기화).

## 7. 리스크 및 유의사항

- **초상권**: 동의 확인 없이는 발행 단계 진입 불가를 시스템으로 강제
- **AI 생성 글**: 항상 사람 검토 후 발행. 강사·기관명 등 고유명사는 시트에서 동기화된 메타데이터에서 주입해 오기 방지
- **네이버 블로그 자동화 의존 금지**: 반자동이 기본 설계, 자동화는 보너스
- **현장과학 통신 환경**: 예미랩(지하) 등 통신 불안정 → 업로드 실패 허용 + 로컬 보관 후 재전송 필수
- **멀티 프로젝트 격리**: URL·쿼리·스토리지 모두 프로젝트 스코프. 클라이언트 A의 데이터가 B 사이트에 노출되는 사고 방지가 최우선
- **진입 장벽**: 성인·시니어 대상 — 로그인 없음 + QR 한 번 동선 원칙 유지
